## 현재 상태 vs. Fabric IQ - 기능 매트릭스

| Fabric IQ 컨셉 | 현재 놀이터 지원 | 갭 |
|-|-|-|
| **엔터티 유형** (이름, 설명) | ✅ 전체 | — |
| **엔터티 유형 이름 제약 조건** (1–26자, 알파벳 및 숫자 + 하이픈/밑줄) | 검증 없음 | 검증 필요 |
| **엔터티 유형 키** (복합형, 하나 이상의 문자열/정수 프로퍼티에서) | ⚠️ 부분 - 프로퍼티당 하나의 `isIdentifier` 논리 값 | 복합 키 모델이 필요함 |
| **인스턴스 디스플레이 이름** (친근한 레이블로 선택한 속성) | 모델링되지 않음 | EntityType에 새로운 필드 |
| **재산권 - 데이터 유형** | ⚠️ 부분 - `string`, `integer`, `decimal`, `date`, `datetime`, `boolean`, `enum` 지원함 | `double`, `long`/`bigint`, `guid`, `time` 누락됨 |
| **속성 - 속성 유형(정적 vs. 시계열)** | 모든 속성은 암묵적으로 정적입니다 | `propertyType: 'static' \| 'timeseries'`가 필요합니다 |
| **재산권 - 이름 제약** (1~26자, 동일 데이터 유형에 대한 모든 엔터티 유형에서 고유함) | 검증 없음 | 엔터티 간 고유성 확인 필요 |
| **재산권 — 의미적 주석** (단위, 메타데이터 속성) | ⚠️ 부분 — `unit` 필드가 존재하며, 일반적인 메타데이터 없음 | 메타데이터 맵을 고려하십시오 |
| **데이터 바인딩 - 정적** (엔터티 유형당 하나, 프로퍼티를 레이크하우스 테이블 열에 매핑) | ⚠️ 부분 - `source`, `table`, `columnMappings`와 함께 `DataBinding`이 존재함 | 바인딩 유형 플래그, 키 매핑이 누락됨 |
| **데이터 바인딩 - 시간 시리즈** (엔터티 유형당 여러 개, 타임스탬프 열, 이벤트하우스 지원) | 지원되지 않음 | 시간 시리즈 바인딩 모델이 필요함 |
| **데이터 바인딩 - 소스 유형** (OneLake lakehouse, Eventhouse, Power BI 의미 모델) | ⚠️ 하드코딩된 `source: string` | 구조화된 소스 유형 enum이 필요함 |
| **관계 유형** (이름, 소스 엔터티, 대상 엔터티) | ✅ 전체 | — |
| **관계 유형 - 데이터 바인딩** (소스/대상 키 열이 있는 링크 테이블) | 모델링되지 않음 | 새로운 `RelationshipBinding` 유형 |
| **관계 유형 - 개수** | ✅ 전체 (`one-to-one`, `one-to-many`, `many-to-one`, `many-to-many`) | — |
| **관계 유형 - 속성** | ✅ 지원됨 (`RelationshipAttribute[]`) | — |
| **관계 유형 - 소스 ≠ 대상 제약** | 검증 없음 | 검증 필요 |
| **형이상학 그래프** (Fabric Graph를 통한 자동 생성된 인스턴스 그래프) | 정적 사이트의 범위 밖 | 정보 제공만 |
| **NL2Ontology 쿼리** | ⚠️ 부분 - NL Query Playground이 존재하지만 데모 전용입니다 | 개선할 수 있음 |
| **버전 관리** | 지원되지 않음(Fabric IQ 아직 이 기능을 지원하지 않음) | 미뤄짐 |
| **세미언틱 모델에서 생성** | Power BI 세미언틱 모델에서 가져오지 않음 | PBIX/TMDL 파서 추가 가능 |
| **거버넌스(규칙, 제약 조건, 데이터 품질 검사)** | 모델링되지 않음 | 제약 조건 모델이 필요함 |

## 1. 엔터티 유형 키 - 복합 키 모델

Fabric IQ 하나 이상의 문자열/정수 속성으로 구성된 **복합 엔터티 유형 키**를 지원합니다. 현재 모델은 각 속성에 간단한 `isIdentifier` 논리 값을 사용하며, 복합 키 개념은 없습니다.

### 1.1 `EntityType`를 `key` 필드로 확장하다
- [ ] `EntityType`에 `key?: string[]`를 추가하세요 - 이 항목은 함께 엔터티 타입의 고유 키를 형성하는 속성 이름들의 순서대로 나열된 목록입니다.
- [ ] 역 호환성을 위해 `isIdentifier` `Property`에 유지하지만 존재할 경우 `key`에서 파생합니다,
- [x] 키에 참여할 수 있는 속성은 `string` 및 `integer`만 가능합니다(이를 검증하세요)

### 1.2 디자이너 UI 업데이트
- [ ] 각 속성별 "Identifier" 확인란을 엔터티 구성 패널의 전용 **Entity Type Key** 섹션으로 대체합니다.
- [ ] 하나 이상의 속성을 키 구성 요소로 선택할 수 있도록 허용합니다(재정렬하려면 드래그)
- [ ] 키가 정의되지 않거나 키 속성이 문자열/정수형이 아닌 유형인 경우 유효성 검사 오류를 표시합니다.

### 1.3 RDF 세리얼라이저/파서 업데이트
- [ ] 복합 키를 사용자 정의 OWL 주석으로 병렬화합니다(예: 프로퍼티 URI의 정렬된 목록이 있는 `ont:entityTypeKey`)
- [ ] RDF에서 복합 키를 다시 파싱하고, 레거시 파일의 경우 `isIdentifier`로 되돌림

### 1.4 업데이트 테스트
- [ ] 복합 키의 왕복 테스트
- [ ] 유효하지 않은 키 구성에 대한 검증 테스트(잘못된 유형, 빈 키)

## 2. 인스턴스 디스플레이 이름

Fabric IQ 사용하면 **인스턴스 디스플레이 이름**으로 프로파일을 선택할 수 있습니다. 이는 하류 경험에서 표시되는 친근한 레이블입니다.

### 2.1 `displayNameProperty`로 `EntityType` 확장하기
- [ ] `EntityType`에 `displayNameProperty?: string`을 추가하세요 - 인스턴스 레이블로 사용되는 속성의 이름입니다.

### 2.2 디자이너 UI 업데이트

### 2.3 그래프 시각화 업데이트
- [ ] `displayNameProperty`가 설정되어 있는 경우, Cytoscape 그래프의 노드 레이블로 사용하십시오(엔터티 유형 이름 대신)
- [ ] 검사기 패널에서 디스플레이 이름 속성을 표시합니다.

### 2.4 RDF 세리얼라이저/파서 업데이트

## 3. 속성 유형 - 정적 vs. 시계열

Fabric IQ **정적** 속성(ID나 위치와 같은 고정된 속성)과 **시간 시리즈** 속성(온도나 압력과 같은 시간이 지남에 따라 변화하는 값, 시간 스탬프 열 포함)을 구분합니다.

### 3.1 `propertyType`로 `Property` 확장하기
- [ ] `Property` 인터페이스에 `propertyType: 'static' | 'timeseries'`를 추가합니다(역향 호환성을 위해 기본값은 `'static'`입니다)

### 3.2 디자이너 UI 업데이트
- [ ] 속성을 추가하거나 편집할 때 "속성 유형" 스위치(정적 / 시간 시리즈)를 추가합니다.
- [ ] 시간 시리즈 속성은 데이터 바인딩에 타임스탬프 열이 필요하다는 것을 설명하는 정보 도구 팝업을 표시해야 합니다.

### 3.3 검사자 패널 업데이트하기
- [ ] 정적 및 시간 시리즈 속성을 구분하는 배지 또는 아이콘을 표시합니다.

### 3.4 RDF 세리얼라이저/파서 업데이트
- [ ] `propertyType`를 사용자 지정 주석 `ont:propertyType`(`"static"` 또는 `"timeseries"`)로 세리얼화합니다.
- [ ] 다시 파싱, 없으면 `"static"`로 기본값으로 설정

### 3.5 그래프/인제 시각화 업데이트
- [ ] 시간 시리즈 속성은 작은 시계 아이콘이나 검사기에서 다른 스타일링으로 렌더링될 수 있습니다.

## 4. 추가 자산 데이터 유형

Fabric IQ 기본 Fabric 데이터 웨어하우스 유형에 매핑되는 데이터 유형을 지원합니다. 현재 Playground는 `string`, `integer`, `decimal`, `date`, `datetime`, `boolean`, `enum` 지원합니다.

### 4.1 누락된 유형을 `Property.type`에 추가합니다.
- [x] `'double'` 추가 - 반지름점형(`xsd:double`로 매핑됨)
- [ ] `'long'` 추가 - 64비트 정수(`xsd:long`에 매핑됨)
- [ ] `'guid'` 추가 - 전역적으로 고유한 식별자(사용자 정의 유형 또는 형식 주석이 있는 `xsd:string`로 매핑됨)
- [ ] `'time'` 추가 - 날짜 없이 하루의 시간 (`xsd:time`로 매핑됨)

### 4.2 디자이너 속성 유형 드롭다운 업데이트
- [ ] 적절한 레이블을 붙인 드롭다운 메뉴에 새로운 유형을 추가하세요.
- [ ] 그룹 유형을 논리적으로 분류하기: 텍스트(문자열, guid) → 숫자(정수, longs, 소수점, 더블) → 날짜/시간(날짜, 시간, datetime) → 기타(부울, enum)

### 4.3 RDF 세리얼라이저/파서 업데이트
- [ ] `XSD_TYPE_MAP`에서 XSD 유형으로의 새로운 유형을 매핑합니다.
- [ ] 각 새로운 유형에 왕복 테스트를 추가합니다.

### 4.4 쿼리 놀이터 업데이트
- [ ] NL 쿼리 엔진이 필터 작업에 대한 새로운 유형을 인식하도록 확인합니다.

## 5. 데이터 바인딩 - 전체 모델

현재의 `DataBinding` 인터페이스는 간단한 매핑(`entityTypeId`, `source`, `table`, `columnMappings`)입니다. Fabric IQ 결합 유형, 구조화된 소스 참조, 키 매핑 및 엔터티 유형당 여러 번 결합을 지원하는 더 풍부한 모델을 가지고 있습니다.

### 5.1 `DataBinding` 인터페이스를 재구축하다
- [ ] `bindingType: 'static' | 'timeseries'` 추가
- [ ] `timestampColumn?: string` 추가 (시간 시리즈 바인딩에 필수)
- [ ] `keyMappings?: Record<string, string>` 추가 - 엔터티 키 프로퍼티를 소스 열에 매핑합니다(일반적인 `columnMappings`와 별개)
- [ ] `source`를 무료 문자열에서 구조화된 유형으로 변경하세요:
  ```ts
  interface DataSource {
    type: 'onelake' | 'eventhouse' | 'semantic-model';
    workspace?: string;
    name: string;    // lakehouse name, eventhouse name, or semantic model name
  }
  ```
- [ ] 엔터티 유형당 여러 개의 바인딩을 지원합니다: 하나의 정적 + N개의 시계열

### 5.2 바인딩 편집기 UI 업데이트
- [ ] 디자이너의 데이터 바인딩 패널을 다음과 같이 재설계합니다.
- 소스 유형 선택기(OneLake / Eventhouse / Semantic Model)
- 결합 유형 선택기(정적 / 시간 시리즈)
- 타임스탬프 열 선택기(시간 시리즈용)
- 키 매핑 섹션(키 프로퍼티를 소스 열과 연결하기 위한 것)
- 열 매핑 섹션(일반 속성에 대한)
- [ ] 엔터티 유형의 바인딩 탭에서 모든 바인딩의 요약 내용을 표시합니다.

### 5.3 검증 규칙
- [ ] 정적 결합: 엔터티 유형당 최대 하나
- [ ] 시간 시리즈 바인딩: 최소한 하나의 정적 바인딩이 있어야 합니다.
- [ ] 시간 시리즈 바인딩: `timestampColumn`이 필요합니다.
- [ ] 시간 시리즈 바인딩의 주요 속성은 정적 바인딩 키와 일치해야 합니다.

### 5.4 RDF 세리얼라이저/파서 업데이트
- [ ] 구조화된 RDF 코멘트 또는 사용자 정의 주석으로 바인딩을 세리얼화합니다(현재 접근 방식은 XML 코멘트를 사용합니다)
- [ ] 더 풍부한 결합 구조를 다시 파싱합니다.

### 5.5 업데이트 테스트
- [ ] 정적 및 시계열 결합에 대한 왕복 테스트
- [ ] 결합 제약 조건에 대한 검증 테스트

## 6. 관계 유형 - 데이터 바인딩

Fabric IQ 관계에는 링크 테이블과 소스 및 대상 엔터티 인스턴스를 식별하는 소스 열을 지정하는 고유한 **데이터 바인딩**. 현재 모델에는 관계 수준 데이터 바인딩 개념이 없습니다.

### 6.1 `RelationshipBinding` 타입 추가
- [ ] 새로운 인터페이스를 생성하세요:
  ```ts
  interface RelationshipBinding {
    source: DataSource;
    table: string;
    sourceEntityColumn: string;  // column matching source entity type key
    targetEntityColumn: string;  // column matching target entity type key
  }
  ```
- [ ] `Relationship` 인터페이스에 `binding?: RelationshipBinding`을 추가합니다.

### 6.2 디자이너 UI 업데이트
- [ ] 관계 구성 패널에 '데이터 바인딩' 섹션을 추가합니다.
- [ ] 소스 데이터 선택기 (워크스페이스 → 레이크하우스 → 테이블)
- [ ] 소스 엔터티 키와 타겟 엔터티 키를 위한 열 선택기
- [ ] 그래프의 관계 모서리에 결합 상태를 표시합니다(결합된 상태와 결합되지 않은 상태)

### 6.3 RDF 세리얼라이저/파서 업데이트

## 7. 엔터티 유형 이름 및 속성 이름 검증

Fabric IQ Playground가 현재 무시하고 있는 엄격한 이름 지정 규칙을 엄격하게 적용합니다.

### 7.1 엔터티 유형 이름 규칙
- [x] 1–26 문자
- [x] 알파벳 및 숫자 문자, 하이픈(`-` 밑줄(`_`)만 가능
- [x] 알파벳과 숫자 문자로 시작하고 끝내야 합니다.
- [x] 디자이너에서 입력 시 유효성 검사(인라인 오류)
- [ ] 가져오기 시 RDF 파서에서 유효성 검사(경고, 차단되지 않음)

### 7.2 속성 이름 규칙
- [x] 1–26 문자
- [x] 엔터티 유형과 동일한 캐릭터 규칙
- [x] 같은 데이터 유형을 공유하는 경우 속성 이름은 **모든 엔터티 유형에서 고유해야 합니다**(예: 두 엔터티 모두 `string ID`를 가질 수 있지만, 하나의 엔터티는 `string ID`이고 다른 엔터티는 `integer ID`일 수는 없음)
- [x] 디자이너에서 검증하기(저장 시 엔터티 간 확인)
- [x] 상호 엔터티 제약 조건을 설명하는 명확한 오류 메시지를 표시합니다.

### 7.3 관계 유형 제약 조건
- [x] ~~소스 및 대상 엔터티 유형은 **별도**여야 합니다(Fabric IQ 자기 참조 관계 없음)~~ **복원됨(#64):** Fabric Ontology *자기 참조 관계를 지원합니다(예: `Employee reportsTo Employee` 따라서 디자이너는 이제 `source == target`을 허용합니다.
- [x] 자기 참조 관계는 그래프 자기 루프(Cytoscape)로 나타납니다.
- [x] "관계 추가"는 단일 엔터티 유형(기본값은 자기 참조)과 함께 작동합니다.

## 8. 거버넌스: 제약 및 데이터 품질 규칙

Fabric IQ 데이터 바인딩 레이어는 스키마 진화 규칙과 데이터 품질 검사(무효성, 범위, 고유성)를 지원합니다. 이는 가장 진보된 격차이며 점진적으로 구현될 수 있습니다.

### 8.1 제약 모델 정의하기
- [ ] `Constraint` 인터페이스를 생성하세요:
  ```ts
  interface PropertyConstraint {
    type: 'required' | 'unique' | 'range' | 'pattern' | 'enum';
    property: string;
    config?: {
      min?: number;
      max?: number;
      pattern?: string;     // regex for pattern constraints
      values?: string[];    // for enum constraints
    };
  }
  ```
- [ ] `EntityType`에 `constraints?: PropertyConstraint[]`를 추가합니다.

### 8.2 제약 조건용 디자이너 UI
- [ ] 엔터티 구성 패널에 "강제 조건" 섹션을 추가합니다.
- [ ] 각 속성에 필요한 고유한 범위 및 패턴 제약 조건을 추가할 수 있도록 허용합니다.
- [ ] 양식의 속성에서 제약 배지를 표시합니다.

### 8.3 검증 엔진
- [ ] 모든 제약 조건이 내부적으로 일관성이 있는지 확인하는 `validateOntology()` 함수를 생성합니다(예: 문자열 속성에 대한 범위 제약 조건이 무효입니다)
- [ ] 내보내기 및 가져오기 전에 검증 실행하기(경고, 차단하지 않음)

### 8.4 RDF 세리얼라이저/파서
- [ ] 제약 조건을 OWL 제한 사항 또는 사용자 정의 주석으로 세리얼화합니다.
- [ ] RDF 다시 파싱하기

## 11. 논리 그래프 & 쿼리 미리보기

Fabric IQ 결합된 데이터에서 인스턴스 그래프를 구축합니다. Playground는 실제 Fabric 데이터에 연결할 수 없지만 실제 Fabric IQ 경험이 어떻게 보이는지 더 잘 나타내기 위해 시각화를 개선할 수 있습니다.

- [ ] "스펙트럼" 보기(엔터티 유형)와 "인스턴스" 보기(엔터티 인스턴스)를 전환합니다.

### 11.2 그래프 쿼리 빌더

RDF/XML 출력이 Fabric IQ 기대하는/생성하는 것과 최대한 일치하도록 합니다.

### 12.1 현재 RDF 출력을 감사하다
- [ ] Playground 세리얼라이저에서 생성된 RDF 사용 가능한 Fabric IQ RDF 수출 또는 샘플과 비교합니다.
- [ ] 네임스페이스 관습, 주석 속성 및 클래스/속성 이름 지정의 문서 차이점

### 12.2 사용자 지정 주석을 정렬하기
- [ ] Fabric IQ 규범(만약 문서화되어 있다면)에 일치하는 주석을 사용하여 다음을 수행합니다. 엔터티 유형 키, 디스플레이 이름, 프로퍼티 유형, 제약 조건, 데이터 바인딩

- [ ] Fabric IQ 특정 형식으로 RDF 내보낸다면, 인자가 이를 왕복으로 처리할 수 있는지 확인하세요.
