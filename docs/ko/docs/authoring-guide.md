#ontology 작성 가이드

그래프 시각화 및 검사기에서 자연어 쿼리 놀이터 및 내장 가능한 위젯에 이르기까지 Ontology Playground 잘 작동하는 엔트로지에이를 만드는 방법.

---

## 빠른 체크리스트

제출하기 전에, 당신의 온톨로지이 다음과 같은 내용을 포함하는지 확인하세요:

- [ ] 명확하고 사람이 읽을 수 있는 **이름**(예: "의료 시스템", "hc_v2"가 아닌 것)
- [ ] 도메인을 한 문장으로 설명하는 **설명**
- [ ] **3-8 엔터티 유형**, 각 엔터티에는 이름, 설명, 아이콘 및 색상이 있습니다.
- [ ] **각 엔터티당 최소 하나의 식별자 속성**(`isIdentifier`로 표시됨)
- [ ] **동사 기반 이름과 Cardinalities**와의 관계
- [ ] 이름, 설명, 카테고리 및 태그가 있는 **metadata.json**

## 1. 논리 구조

이 프로젝트의 온톨로지은 네 가지 부분으로 나뉩니다:

```
Ontology
├── name            "Fourth Coffee"
├── description     "A coffee shop chain with suppliers, products, stores…"
├── entityTypes[]   The concepts in your domain
└── relationships[] How those concepts connect
```

### 엔터티 유형

각 엔터티 유형은 현실 세계의 개념을 나타냅니다. 이를 데이터베이스의 테이블이나 UML 다이어그램의 클래스라고 생각하세요.

```
EntityType
├── id              "customer"          (URL-safe slug, unique within the ontology)
├── name            "Customer"          (human-readable, singular noun)
├── description     "A person who purchases coffee products"
├── icon            "👤"               (emoji)
├── color           "#0078D4"           (hex color for the graph node)
└── properties[]
    ├── { name: "customerId", type: "string", isIdentifier: true }
    ├── { name: "name",       type: "string" }
    ├── { name: "email",      type: "string" }
    └── { name: "loyaltyTier", type: "enum", values: ["Bronze","Silver","Gold"] }
```

### 관계들

각 관계는 두 엔터티 유형 사이의 방향성 링크입니다.

```
Relationship
├── id              "places"            (URL-safe slug, unique)
├── name            "places"            (a verb — "places", "contains", "teaches")
├── from            "customer"          (source entity ID)
├── to              "order"             (target entity ID)
├── cardinality     "one-to-many"       (one-to-one | one-to-many | many-to-one | many-to-many)
├── description     "A customer places one or more orders"
└── attributes[]    (optional — properties on the relationship itself)
    └── { name: "quantity", type: "integer" }
```

## 2. 필드별 가이드

온톨로지에 있는 모든 필드는 앱의 어느 곳에서나 사용됩니다. 여기 정확히 어디와 무엇이 좋은 가치를 만드는지입니다.

### 엔터티 필드

| 필드 | 사용처 | 모범 사례 |
|-------|---------|---------------|
| `id` | 내부 그래프 ID, URL 라우팅, 쿼리 엔진 매칭 | 소문자 슬러그, 공백 없음: `customer`, `work-order` |
| `name` | 그래프 노드 레이블, 검사기, 쿼리 제안, 검색 | **단수 명사**, 대문자 제목: "고객", "작업 주문" |
| `description` | 검사기 패널, 쿼리 결과, 검사기 삽입 | 개념을 설명하는 한 문장. 기사로 시작: "어떤 사람은…" |
| `icon` | 그래프 노드 레이블, 검사기 머리글, 쿼리 결과 | 개념을 나타내는 단일 이모티콘: 👤 🛒 🏥 🏭 |
| `color` | 그래프 노드 배경, 검사기 강조 표시 | 해시색. 일관성을 위해 Microsoft 팔레트를 사용하십시오(아래 참조) |
| `properties` | 검사자 세부 정보, 쿼리 제안, RDF 출력 | 엔터티당 3~8개의 속성. 흥미로운 쿼리를 위한 유형 혼합 |

### 자산 필드

| 필드 | 사용처 | 모범 사례 |
|-------|---------|---------------|
| `name` | 검사관 속성 목록, 쿼리 제안, RDF 출력 | camelCase, 설명적: `loyaltyTier`, `lt` 또는 `field3`이 아닌 것 |
| `type` | 검사자 유형 배지, RDF XSD 매핑 | 다음 중 하나: `string`, `integer`, `decimal`, `date`, `datetime`, `boolean`, `enum` |
| `isIdentifier` | 검사기, 검증, RDF 출력에서 🔑 배지 | 각 엔터티에 정확히 **하나**의 속성이 식별자여야 합니다 |
| `description` | 검사자(있는 경우) | 선택 사항. 명확하지 않은 속성에 유용함 |
| `unit` | 검사자(만약 존재한다면) | 선택 사항. 예를 들어, `"USD"`, `"kg"`, `"minutes"` |
| `values` | 검사기 enum 디스플레이 | `type`가 `enum` 때 필수입니다. 허용되는 값의 배열 |

### 관계 필드

| 필드 | 사용처 | 모범 사례 |
|-------|---------|---------------|
| `id` | 내부 에지 ID, 쿼리 엔진 일치 | 소문자 슬러그: `places`, `sourced-from` |
| `name` | 그래프 에지 레이블, 쿼리 결과, 검사기 | **방향에 대한 설명을 하는 활성 동사**: "장소", "강사", "감시" |
| `from` / `to` | 그래프 에지 소스/타겟 | 기존 엔터티 `id` 값과 일치해야 함 |
| `cardinality` | 검사자, 쿼리 결과, RDF 출력 | 현실 세계의 다중성을 생각해 보세요. 고객이 *많은* 주문을 내립니다 → `one-to-many` |
| `description` | 검사 패널 | 한 문장: "고객이 하나 이상의 주문을 하였습니다." |
| `attributes` | 검사기, RDF 출력 | 선택 사항. 엔터티가 아닌 *관계*,에 속하는 속성에 사용(예: 주문→제품 링크의 `quantity`) |

## 3. 쿼리 놀이터가 당신의 논리학을 사용하는 방식

자연어 쿼리 놀이터는 제안과 당신의 온톨로지의 필드에 따라 쿼리를 생성하고 처리합니다. 이를 이해하는 것은 좋은 쿼리 경험을 생성하는 온톨로지을 작성하는 데 도움이 됩니다.

### 자동 생성된 제안

이 앱은 당신의 온톨로지에서 최대 6개의 샘플 쿼리를 생성합니다:

| 패턴 | 출처 | 예시 |
|---------|--------|---------|
| "모든 **X**를 보여줘" | 첫 번째 엔터티의 `name`(복수형) | "모든 고객을 보여줘" |
| "모든 **Ys** 나열" | 두 번째 엔터티의 `name` | "모든 주문 나열" |
| "**프로**에 의해 **X** 표시" | 식별자 없는 문자열 속성 | "고객을 **리워드 등급**별로 표시" |
| "**X**는 **Y**와 어떻게 연결됩니까?" | 첫 번째 관계의 `from`/`to` 엔터티 | "고객은 어떻게 주문을 연결합니까?" |

**팁:** 만약 귀하의 엔터티 이름이 깨끗한 명사이고 귀하의 속성이 설명적인 이름이 있다면, 자동 제안은 자연스럽게 읽힐 것입니다. 만약 귀하의 엔터티가 "tbl_cust"라고 명명된다면, 제안은 "모든 tbl_cust를 보여줘"가 됩니다. 좋지 않습니다.

### 쿼리 일치

사용자가 질문을 입력할 때, 엔진은 다음과 같이 일치시킵니다.

- **엔터티 이름** - "모든 **고객**을 보여줘" → 고객 엔터티를 강조 표시합니다.
- **재산명** - "**고객 충성도 등급**별로 고객 표시" → 고객 강조 표시
- **관계 동사** - "고객은 어떻게 **주문**과 **연결**합니까?" → 관계를 강조합니다.
- **수량 조회** - "몇 개의 **주문**" → 엔터티 수량을 표시합니다.
- **개념적 질문** - "엔터티 유형이란 무엇인가?" → 모든 엔트로지에 적용 가능

당신의 이름과 설명이 더 좋을수록 검색 결과도 더 풍부해집니다.

### 쿼리 결과에서 나타나는 내용

사용자가 엔터티를 조회할 때, 그들은 다음과 같은 것을 보게 됩니다.

```
**Customer** 👤
A person who purchases coffee products from our stores

**Properties:**
• customerId (string) 🔑
• name (string)
• email (string)
• loyaltyTier (enum)
```

이것은 `entity.name`, `entity.icon`, `entity.description`, 그리고 `entity.properties`에서 구성됩니다. 만약 이것 중 하나가 비어 있다면, 결과는 불완전해 보입니다.

## 4. 그래프 시각화가 당신의 온톨로지론을 어떻게 사용하는가

### 노드 모양

각 엔터티는 Cytoscape 그래프의 원형 노드로 변합니다:

- **라벨** = `icon` + `name` (예: "👤 고객")
- **배경색** = `color`
- **크기** = 일정한 크기(60px), 선택 시 확대됨

**차별화된 색상**을 사용하여 엔터티를 한눈에 쉽게 구분할 수 있도록 하십시오. 공식 엔톨로지에는 이 마이크로소프트 친화적인 팔레트가 사용됩니다.

| 색상 | 해시 코드 | 일반적인 사용 |
|-------|-----|-------------|
| 파란색 | `#0078D4` | 주요 엔터티(고객, 환자) |
| 녹색 | `#107C10` | 보조 엔터티(제품, 제공자) |
| 주황색 | `#D83B01` | 액션 엔터티(주문, 예약) |
| 보라색 | `#5C2D91` | 지원 엔터티(매장, 부서) |
| 파란색 | `#008272` | 인프라스트럭처(기계, 센서) |
| 시안 | `#00A9E0` | 금융 (계좌, 대출) |
| 노란색 | `#FFB900` | 경고/특별(품질검증) |
| 빨간색 | `#E81123` | 중증(진단, 처방) |

### 모서리 모양

각 관계는 지시된 에지에 변합니다:

- **라벨** = `name` (예: "장소")
- **화살표**는 `from` → `to`에서 가리킵니다.

**팁:** 관계 이름을 짧게 유지하세요(1~2단어). 긴 이름은 그래프에서 중복됩니다. "sourcedFrom"는 "is sourced from supplier"보다 낫습니다.

### 레이아웃

이 그래프는 `fcose` 힘 지시 레이아웃을 사용합니다. 그것은 다음과 함께 가장 잘 작동합니다:

- **5~8개 엔터티** - 적은 엔터티수는 흩어져 보이지만, 많은 엔터티수는 붐비게 됩니다.
- **연결된 그래프** - 모든 엔터티에는 적어도 하나의 관계가 있어야 합니다.
- **합리적인 밀도** - 엔터티당 평균 1~2개의 관계

## 5. 인포그램밍 위젯이 당신의 논리 구조를 어떻게 사용하는가

인바운드 가능한 위젯은 동일한 그래프와 인스펙터를 독립형 컨테이너에 렌더링합니다. 위에서 언급한 모든 내용이 적용되며, 추가적으로:

- **RDF 소스 탭** - 위젯은 your ontology를 RDF/XML 표시합니다. 깨끗한 이름은 읽을 수 있는 RDF 생성합니다.
- **컴팩트 검사기** - 하단 오버레이 공간이 제한되어 있으므로 설명을 간결하게 유지하세요(단락이 아닌 문장 하나)
- **검색/필터 없음** - 인포그래픽에는 검색창이 없으므로 사용자가 노드를 클릭하는 데 의존합니다. 엔터티 아이콘과 색상이 명확하게 구분되어 있는지 확인하십시오.

## 6. 카탈로그용 메타데이터.json

모든 카탈로그 오노토미는 `.rdf` 파일과 함께 `metadata.json`이 필요합니다:

```json
{
  "name": "Fourth Coffee",
  "description": "A sample ontology representing a coffee shop chain with suppliers, products, stores, customers, and orders.",
  "icon": "☕",
  "category": "retail",
  "tags": ["coffee", "supply-chain", "fabric-iq"],
  "author": "your-github-username"
}
```

| 필드 | 필수 | 참고 사항 |
|-------|----------|-------|
| `name` | 예 | 온톨로지의 `name` 필드와 동일 |
| `description` | 예 | 갤러리 카드에 대한 짧은 설명 |
| `category` | 예 | 다음 중 하나: `retail`, `healthcare`, `finance`, `manufacturing`, `education`, `technology`, `general`, `fibo` |
| `icon` | 아니요 | 갤러리 카드에 대한 이모티콘. 기본적으로 일반 아이콘으로 설정됩니다 |
| `tags` | 아니요 | 필터링용 문자열 배열. 소문자로 유지하세요 |
| `author` | 아니요 | 당신의 GitHub 사용자 이름입니다. 갤러리에서 표시됩니다 |

## 7. 흔한 실수들

| 실수 | 문제 | 해결 |
|---------|---------|-----|
| 식별자 프로퍼티 없음 | 검증 실패, Fabric IQ 엔터티를 해결할 수 없음 | 엔터티당 하나의 프로퍼티에 `isIdentifier: true`를 추가함 |
| 일반 엔터티 이름("상품", "물건") | 형편없는 쿼리 제안 및 검색 결과 | 도메인별 명사("제품", "처방") 사용 |
| 설명이 누락됨 | 검사 및 쿼리 결과가 공백으로 표시됨 | 모든 엔터티에 한 문장 설명을 추가함 |
| 중복 ID | 빌드 스크립트가 온톨로지를 거부합니다 | `entity1`, `entity2` 대신 `customer`, `order`처럼 고유한 슬러그를 사용합니다. |
| 관계 없음 | 그래프는 연결되지 않은 노드를 보여주며, 흥미로운 쿼리는 없음 | 엔터티들 사이에 적어도 하나의 관계를 추가하세요 |
| 명사 기반 관계 이름("소유권") | 그래프 에지 레이블로 이상하게 읽힌다 | 동사 사용: "소유하다", "놓다", "교육하다" |
| 너무 많은 엔터티 (>10) | 그래프 레이아웃이 복잡해진다 | 5~8개의 핵심 개념에 집중하고, 더 큰 도메인을 분할한다 |
| 모든 엔터티에 동일한 색상 | 그래프에서 노드를 구분할 수 없음 | 위의 팔레트에서 고유한 색상을 할당함 |

## 8. 단계별 안내: 처음부터 온톨로지론을 만드는 방법

**도서관 시스템**을 단계별로 모델링해 보자.

### 단계 1: 당신의 엔터티를 식별하세요

도메인의 핵심 개념에 대해 생각해 보세요:
- **책** - 카탈로그의 제목
- **저자** - 책을 쓰는 사람
- **회원** - 도서관 이용자
- **대출** - 회원이 빌린 책
- **지점** - 물리적 도서관 위치

### 단계 2: 속성을 정의하다

각 엔터티에 대해 속성을 나열합니다.

| 엔터티 | 속성 |
|--------|-----------|
| 책 | `isbn` (문자열, `title` (문자열), `genre` (정수형 enum), `pageCount` (정수형), `publishedDate` (날짜) |
| 저자 | `authorId` (문자열, `name` (문자열), `nationality` (문자열) |
| 회원 | `memberId` (문자열, `name` (문자열), `email` (문자열), `memberSince` (날짜) |
| 대출 | `loanId` (문자열, `loanDate` (날짜), `dueDate` (날짜), `returned` (부울형) |
| 지점 | `branchId` (문자열, `name` (문자열), `city` (문자열), `capacity` (정수) |

### 단계 3: 관계를 정의하기

이러한 개념들은 어떻게 연결되어 있나요?

| 관계 |에서→까지 | Cardinality | 왜 |
|-------------|-----------|-------------|-----|
| `writtenBy` | 책 → 작가 | 다중-일대 | 각 책에는 하나의 주요 작가가 있습니다 |
| `borrows` | 회원 → 대출 | 일대다수 | 한 회원은 여러 대출을 가질 수 있다 |
| `loanOf` | 대출 → 책 | 다중-일대 | 각 대출은 한 권의 책에 대한 것입니다 |
| `locatedAt` | 책 → 지점 | 다대일 | 책 사본은 지점에 속합니다 |
| `registeredAt` | 회원 → 지점 | 다대일 | 회원들은 지점에 등록합니다 |

## 10. 참조: 개수

| 개수 | 의미 | 예시 |
|-------------|---------|---------|
| `one-to-one` | 각 A는 정확히 하나의 B로 매핑되며 , 반대의 경우도 마찬가지입니다 | 직원 ↔ 직원 배지 |
| `one-to-many` | 각 A는 많은 B로 매핑됨 | 고객 → 주문 |
| `many-to-one` | 많은 것이 하나 B로 매핑됨 | 주문 → 매장 |
| `many-to-many` | Many As ↔ Many Bs (종종 교차 엔터티가 필요함) | 학생 ↔ 과정 (등록을 통해) |
