## 학업 기록의 기초

학업 기록은 핵심 질문 주위에 둘러싸여 있습니다: *어떤 학생들이 어떤 과목을 수강하고, 그리고 그들은 어떻게 성적을 내고 있나요?* 세 가지 단체가 이 질문에 답합니다:

- **학생** - 누가 배우고 있나요?
- **과정** - 무엇을 가르치고 있는가?
- **등록** - 학생을 성적과 함께 강좌에 연결하는 기록

## 엔터티 정의하기

### 학생

| 재산 | 유형 | 식별자? |
|---|---|---|
| `studentId` | 문자열 | ✓ |
| `name` | 문자열 | |
| `gpa` | 플로트 | |
| `enrollmentYear` | 정수 | |
| `major` | 문자열 | |

`gpa` 속성은 플로트입니다. 학점 평균은 0.0에서 4.0까지입니다. 이 집계 지표는 학업 성취도 쿼리 및 명예 목록 계산에 사용됩니다.

### 과정

| 재산 | 유형 | 식별자? |
|---|---|---|
| `courseId` | 문자열 | ✓ |
| `title` | 문자열 | |
| `credits` | 정수 | |
| `level` | 문자열 | |
| `maxEnrollment` | 정수 | |

`level` 속성(100, 200, 300, 400)은 과정의 난이도와 사전 요건을 나타냅니다. `maxEnrollment` 정수는 수용 인원 계획을 가능하게 합니다.

### 등록

| 재산 | 유형 | 식별자? |
|---|---|---|
| `enrollmentId` | 문자열 | ✓ |
| `semester` | 문자열 | |
| `grade` | 문자열 | |
| `enrollDate` | 날짜 | |
| `status` | 문자열 | |

등록은 **교차 엔터티**. 이는 학생과 과정에 추가적인 맥락(학년, 학기, 상태)을 연결하기 위해 특별히 존재합니다.

## 관계들

- **enrolls_in** — `Student` → `Enrollment` (일대다) 학생은 학기별로 여러 등록이 있습니다.

- **for_course** — `Enrollment` → `Course` (여러개에서 하나로) 각 등록은 하나의 특정 과정에 대한 것입니다.

> **접합 엔터티 패턴:** 두 엔터티가 속성과 많은 대 많은 관계를 가질 때, 당신은 접합 엔터티를 생성합니다. 학생은 여러 과목을 수강합니다. 과목에는 여러 학생이 있습니다. 등록은 그들과 사이에 위치하며, 성적, 학기 및 상태를 담고 있습니다. 이것은 오노토미 설계에서 가장 일반적인 패턴 중 하나입니다.

## 우리가 배운 것

- **접합 엔터티**(등록)는 속성과 관련된 다대다 관계를 해결합니다.
- **플로트 속성**(GPA)은 집계 계산과 임계를 가능하게 합니다.
- **정수 속성**(학점, maxEnrollment)은 용량 및 워크로드 계획을 가능하게 합니다.
- 학업 핵심은 학생 → 등록 → 과정 순으로 이어집니다.

```quiz
Q: 왜 등록은 학생-과목 관계가 아닌 별도의 엔터티로 모델링되는가?
- 그래프가 더 많은 노드를 가지도록 하기 위해
- 등록은 학생이나 과정에 속하지 않는 고유한 속성(학년, 학기, 상태)을 가지고 있기 때문에 [correct]
- 엔터티론은 최소 세 개의 엔터티를 필요로 한다.
- 엔터티 간의 직접적인 관계는 허용되지 않습니다.
> 직선적인 학생-과목 관계는 성적, 학기 또는 상태 정보를 전달할 수 없습니다. 교차 엔터티 패턴은 관계 자체에 대해 일류 엔터티를 생성하여 "이 학생이 이번 학기에 이 과목에서 어떤 성적을 받았습니까?"와 같은 쿼리를 수행할 수 있습니다. 이는 두 끝점뿐만 아니라 연결에 속성이 필요합니다.
```

다음으로, 우리는 교수님을 추가하여 교수진의 업무 분배 상황을 추적할 것입니다.
