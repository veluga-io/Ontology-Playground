## 활동 추적

거래내역이 없는 계좌는 그냥 정적인 잔액일 뿐입니다. **거래**를 추가하면 모든 구매, 입금, 이체, 수수료 등 자금 흐름을 포착할 수 있습니다.

이를 통해 다음과 같은 질문이 가능해집니다.
- "이 고객은 지난 달에 식당에서 얼마를 썼습니까?"
- "어떤 계좌에 특이한 거래 패턴이 있나요?"
- "계좌 종류별 평균 거래 금액은 얼마인가요?"

## 거래 주체

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `transactionId` | 문자열 | `amount` |
| `type` | 십진수(USD) | |
| `timestamp` | 문자열 | |
| `merchant` | 날짜/시간 | |
| ✓ | 문자열 | |

금융 거래에는 정확성이 필요하기 때문에 `timestamp`는 날짜/시간(날짜만이 아님)입니다. 사기 감지를 위해 오후 2시 30분의 구매와 오후 2시 31분의 구매는 다릅니다.

`merchant` 속성은 거래가 발생한 위치를 캡처하여 지출 범주 분석에 유용합니다.

## 새로운 관계

- **has_transaction** — `Account` → `Transaction`(일대다) 각 계정에는 시간이 지남에 따라 많은 트랜잭션이 있지만 각 트랜잭션은 하나의 계정에 속합니다.

이는 소유권 체인을 확장합니다: `Customer → Account → Transaction`.

## 성장하는 그래프

<ontology-embed id="official/finance-step-2" diff="official/finance-step-1" height="400px"></ontology-embed>

*트랜잭션은 액티비티 레이어를 추가합니다. 소유권 체인이 성장합니다: 고객 → 계정 → 거래.*

## 우리가 배운 것

- **날짜/시간 정밀도**는 재무 및 규정 준수 시나리오에 중요합니다.
- **소유권 체인**(고객 → 계정 → 거래)을 통해 드릴다운 쿼리 가능
- `merchant` 속성은 판매자 엔터티를 추가하지 않고도 지출 분석을 엽니다.
- 각각의 새로운 엔터티는 답변할 수 있는 질문을 심화시킵니다.

```quiz
Q: Transaction이 날짜 대신 타임스탬프에 날짜/시간 유형을 사용하는 이유는 무엇입니까?
- 날짜/시간은 모든 시간 기반 필드의 기본 속성 유형입니다.
- 금융 거래에는 사기 탐지 및 감사 추적을 위한 정확한 시간이 필요합니다. [correct]
- 최신 온톨로지에서는 날짜 유형이 더 이상 사용되지 않습니다.
- 날짜 시간은 날짜보다 적은 저장 공간을 사용합니다.
> 금융 규정 준수 및 사기 탐지에는 정확한 타임스탬프가 필요합니다. 같은 날짜에 두 건의 거래가 이루어졌지만 몇 분 간격으로 이루어졌을 경우 사기 패턴을 나타낼 수 있습니다. Datetime은 날짜와 시간을 모두 캡처하여 필요한 정밀도를 제공합니다.
```
