## 주문이 이루어지는 곳

지금까지 우리는 *누가* *무엇을* 주문하는지 알지만 *어디*를 주문하지는 않습니다. Fourth Coffee는 여러 도시에 걸쳐 매장을 운영하고 있으며 각 주문은 특정 매장에서 처리됩니다.

**Store** 엔터티를 추가하면 다음과 같은 위치 기반 질문에 답할 수 있습니다.
- "주문이 가장 많은 매장은 어디인가요?"
- "도시별 평균 주문 금액은 얼마인가요?"
- "주문량에 따라 어떤 매장에 직원이 더 필요한가요?"

## 매장 엔터티

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `storeId` | 문자열 | `name` |
| `city` | 문자열 | |
| `state` | 문자열 | |
| `openDate` | 문자열 | |
| `capacity` | 날짜 | |
| ✓ | 정수 | |

`capacity` 속성(좌석 수용력)은 **정수**로 운영 계획에 유용합니다. `city` 및 `state` 속성은 전체 주소 계층 구조의 복잡성 없이 지리적 컨텍스트를 제공합니다.

## 새로운 관계

- **processedAt** — `Order` → `Store`(다대일) 각 주문은 정확히 하나의 매장에서 처리되지만, 매장은 많은 주문을 처리합니다.

> **디자인 노트:** 이는 다대일 관계입니다. 많은 주문이 하나의 매장에 매핑됩니다. 이는 "속함" 또는 "발생 위치" 관계에 대한 가장 일반적인 카디널리티 패턴입니다.

## 우리가 배운 것

- **다대일 관계** 모델은 "속함" 또는 "위치" 패턴에 속함
- **정수 속성**은 셀 수 있는 수량(용량, 층, 좌석)에 적합합니다.
- 하나의 엔터티를 추가하면 위치 기반 쿼리의 전체 범주가 열립니다.
- `diff` 보기는 변경된 내용을 정확하게 표시하므로 온톨로지 진화를 쉽게 추적할 수 있습니다.

```quiz
Q: Order와 Store 간의 "processedAt" 관계에는 어떤 카디널리티가 있어야 합니까?
- 일대일 — 각 매장에는 정확히 하나의 주문이 있습니다.
- 일대다 — 각 주문이 여러 매장에서 처리됩니다.
- 다대일 — 한 매장에서 많은 주문이 처리됩니다. [correct]
- 다대다 - 여러 매장에서 동시에 주문을 처리할 수 있습니다.
> 각 주문은 정확히 한 매장 위치에서 처리되지만, 매장은 하루 종일 많은 주문을 처리합니다. Order의 관점에서 볼 때 이는 다대일입니다.
```
