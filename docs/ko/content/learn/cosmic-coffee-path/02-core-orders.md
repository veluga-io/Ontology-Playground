## 기초

모든 상거래 시스템은 세 가지 핵심 개념으로 시작됩니다.

- **고객** — 누가 구매하나요?
- **주문** — 어떤 거래가 발생했나요?
- **제품** — 무엇을 구매했습니까?

이 세 가지 엔터티 유형은 Fourth Coffee 온톨로지의 핵심을 형성합니다. 나중에 추가하는 모든 항목은 다시 연결됩니다.

## 엔터티 정의

### 고객

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `customerId` | 문자열 | `name` |
| `email` | 문자열 | |
| `loyaltyTier` | 문자열 | |
| `joinDate` | 열거형(브론즈, 실버, 골드, 플래티넘) | |
| `totalSpend` | 날짜 | |
| ✓ | 십진수(USD) | |

`customerId`는 각 고객을 고유하게 식별합니다. `loyaltyTier`는 열거형을 사용하여 값을 유효한 계층으로 제한합니다. 이는 다운스트림 분석에서 데이터 품질 문제를 방지합니다.

### 주문

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `orderId` | 문자열 | `timestamp` |
| `total` | 날짜/시간 | |
| `status` | 십진수(USD) | |
| `paymentMethod` | enum(보류 중, 준비 중, 준비됨, 완료됨, 취소됨) | |
| ✓ | enum (카드, 현금, 모바일, 상품권) | |

### 제품

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `productId` | 문자열 | `name` |
| `category` | 문자열 | |
| `price` | enum (에스프레소, 양조, 콜드브루, 차, 식품, 상품) | |
| `origin` | 십진수(USD) | |
| `isOrganic` | 문자열 | |
| ✓ | 부울 | |

`isOrganic` 플래그는 부울이므로 나중에 필터링 및 규정 준수 쿼리에 유용합니다.

## 관계로 연결하기

엔터티 자체는 단지 격리된 테이블일 뿐입니다. **관계**를 그래프로 변환합니다.

- **장소** — `Customer` → `Order`(일대다) 각 고객은 여러 주문을 할 수 있지만 각 주문은 한 고객에게 속합니다.

- **포함** — `Order` → `Product`(다대다) 주문에는 여러 제품이 포함될 수 있으며 제품은 여러 주문에 나타날 수 있습니다.

## 우리가 배운 것

- 모든 엔터티에는 **식별자 속성**, 즉 고유 키가 필요합니다.
- **열거형 속성**은 값을 유효한 옵션으로 제한합니다.
- **부울 속성**으로 간단한 필터링 가능
- **카디널리티**(일대다 대 다대다)는 엔터티의 관계 방식을 결정합니다.

```quiz
Q: 주문과 제품 간의 "포함" 관계가 일대다가 아닌 다대다로 설정된 이유는 무엇입니까?
- 각 주문에는 하나의 제품만 포함될 수 있습니다.
- 제품은 한 번에 하나의 주문에만 표시될 수 있습니다.
- 하나의 주문에 여러 제품이 포함될 수 있으며 하나의 제품이 여러 주문에 나타날 수 있습니다. [correct]
- 다대다 관계는 항상 기본 관계 유형입니다.
> 주문에는 일반적으로 여러 제품(라떼, 머핀, 콩 한 봉지)이 포함되며 각 제품은 다양한 주문에 걸쳐 나타납니다. 이러한 양방향 다중성을 위해서는 다대다가 필요합니다.
```
