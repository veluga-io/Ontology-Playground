## 품질 루프 닫기

부품이 생산된다고 제조가 끝나는 것이 아닙니다. 부품을 검사해야 합니다. **품질 검사**는 부품이 사양을 충족하는지 확인하여 생산 주기를 종료합니다.

## 품질 검사 엔터티

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `checkId` | 문자열 | `inspector` |
| `checkDate` | 문자열 | |
| `passed` | 날짜 | |
| `defectCode` | 부울 | |
| ✓ | 문자열 | |

`passed` 부울은 부품 배송 또는 재작업 여부를 결정하는 중요한 속성입니다. `defectCode` 속성은 근본 원인 분석을 위해 실패를 분류합니다.

## 새로운 관계

- **검사** — `Quality-Check` → `Part`(다대일) 각 품질 검사는 특정 부품을 검사합니다. 부품은 여러 검사(초기 검사, 재작업 후 재검사)를 거칠 수 있습니다.

> **피드백 루프:** 품질 검사에 실패하면 생산 체인이 반전됩니다: `Quality-Check (passed=false) → Part → Work-Order → Machine`. 이 피드백 루프는 스마트 팩토리가 문제가 있는 기계를 식별하고 시간이 지남에 따라 생산 품질을 향상시키는 방법입니다.

*완전한 스마트 제조 온톨로지: 5개 엔터티, 5개 관계. 품질 검사는 검사부터 생산까지 피드백 루프를 닫습니다.*

## 완전한 모델이 가능하게 하는 것

| 질문 | 그래프 경로 |
|---|---|
| 검사에 실패한 부품을 생산하는 기계는 무엇입니까? | 기계 → 부품 ← 품질검사(통과=false) |
| 결함이 있는 부품이 생산되었을 때 어떤 센서가 이상했습니까? | 센서 → 기계 → 부품 ← 품질검사(통과=false) |
| 작업지시 우선순위별 불량률은 어떻게 되나요? | 작업지시(우선순위) → 부품 ← 품질검사 |
| 어떤 부분을 재검사해야 합니까? | 부분 ← 품질 확인(통과=false, 개수 > 1) |

## GQL 쿼리 예

센서 이상과 품질 문제의 상관관계:

```gql
MATCH (s:Sensor)-[:monitors]->(m:Machine)-[:has_part]->(p:Part)<-[:inspects]-(qc:QualityCheck)
WHERE s.lastReading > s.threshold AND qc.passed = false
RETURN m.name, s.type, s.lastReading, p.name, qc.defectCode
```

## 우리가 만든 것

| 단계 | 추가된 엔터티 | 누적 | 핵심 개념 |
|---|---|---|---|
| 1 | 기계, 센서 | 2 | IoT 계층 구조, 원격 측정 |
| 2 | 작업 주문, 부품 | 4 | 생산 체인, 공차 |
| 3 | 품질검사 | 5 | 피드백 루프, 검사 |

## 주요 내용

1. **IoT 계층 구조**는 원격 측정 집계를 위해 시스템 아래에 센서를 구성합니다.
2. **생산 체인**은 일정 관리 엔터티를 통해 장비를 출력에 연결합니다.
3. **품질 피드백 루프**를 통해 생산 체인 전반에 걸쳐 근본 원인 분석이 가능합니다.
4. **임계값 기반 경고**로 예측 유지 관리 지원
5. **부울 속성**(통과)은 워크플로에서 명확한 결정 지점을 만듭니다.

```quiz
Q: 품질 검사는 제조 온톨로지에서 피드백 루프를 어떻게 생성합니까?
- 기계에 직접 연결됩니다.
- 실패한 검사는 부품 → 작업 주문 → 기계를 통해 역추적되어 결함의 원인을 식별합니다. [correct]
- 센서 엔터티로 루프백됩니다.
- 품질 검사로 인해 피드백 루프가 생성되지 않습니다.
> 품질 검사에 실패하면 품질 검사 → 부품 → 작업 주문 → 기계 경로를 통해 결함의 원인을 추적합니다. 이 피드백 루프는 결함이 있는 부품을 생산하는 기계, 작업 주문 또는 조건을 식별하는 스마트 제조의 지속적인 개선에 기본입니다.
```
