## 인간을 위한 이름 짓기

가장 중요한 디자인 결정은 이름 지정입니다. 엔터티 유형과 속성은 사람과 기계 모두에서 읽혀집니다. 이름이 명확하면 자연어 쿼리가 더 정확해집니다.

**해야 할 일:**
- 엔터티 유형에는 단수 명사 사용: `Customer`, `Product`, `Order`
- 속성에 camelCase를 사용하세요: `firstName`, `totalAmount`, `createdDate`
- 관계에 동사구를 사용하세요: `placedBy`, `worksAt`, `contains`

**하지 마세요:**
- 내부 테이블 이름 사용: `tbl_cust_v2`, `DIM_PRODUCT`
- 약어: `qty`, `amt`, `dt` — 철자를 입력하세요.
- 일반 이름 사용: `Item`, `Record`, `Thing`

## 하나의 개체, 하나의 개념

각 엔터티 유형은 **단일 비즈니스 개념**을 나타내야 합니다. 관련되지 않은 속성을 추가하는 경우 엔터티를 분할해야 할 수도 있습니다.

**안티패턴:** `Person`, `salary`, `patientId` 및 `courseGrade`가 포함된 `accountBalance` 엔터티 — 이는 4가지 서로 다른 개념(직원, 환자, 학생, 고객)을 하나로 강제합니다.

**더 나은 점:** 별도의 엔터티 유형을 만들고 필요한 경우 연결합니다. `Person`는 `Employee` 레코드, `Patient` 레코드 등에 연결할 수 있습니다.

## 식별자를 신중하게 선택하세요

식별자 속성은 인스턴스의 계산, 그룹화 및 결합 방법을 결정합니다. 좋은 식별자는 다음과 같습니다.

- 모든 인스턴스에서 **고유**
- **안정적** — 시간이 지나도 변하지 않습니다.
- **의미 있음** — 내부 자동 증가가 아닌 비즈니스 키를 사용하는 것이 좋습니다.

예: 도서의 경우 `isbn`, 사용자의 경우 `email`, 주문의 경우 `orderId`.

복합 식별자(함께 키를 형성하는 여러 필드) 사용을 피하세요. 대부분의 온톨로지 도구는 엔터티당 단일 식별자를 기대합니다.

## 외래 키가 아닌 모델 관계

관계형 데이터베이스에서는 외래 키를 사용하여 테이블을 연결합니다. 온톨로지에서는 명시적인 의미와 함께 **명명된 관계**를 사용합니다.

| 관계형 | 온톨로지 |
|------------|----------|
| `orders.customer_id → customers.id` | `Order` → `placedBy` → `Customer` |
| `order_items.product_id → products.id` | `OrderItem` → `contains` → `Product` |

**이름** 관계는 매우 중요합니다. 이는 쿼리 엔진(및 인간)에게 연결의 의미를 알려줍니다. "placedBy"는 `fk_cust_id`라는 열보다 훨씬 더 명확합니다.

## 올바른 카디널리티 확보

잘못된 카디널리티는 잘못된 집계로 이어집니다. 스스로에게 물어보세요. "A의 한 인스턴스에 대해 B의 인스턴스는 몇 개나 있을 수 있나요?"

- 고객은 **다수** 주문 → 일대다 주문 가능
- **one** 매장에서 주문 → 다대일 주문
- 학생은 **다** 과목을 수강할 수 있고, 과목은 **다** 학생 → 다대다로 구성됩니다.

*헬스케어 온톨로지는 카디널리티에 대한 좋은 연구입니다. 환자는 많은 약속을 갖고 있지만 각 약속에는 하나의 공급자가 있습니다. 진단은 한 명의 환자에게 속하지만 여러 처방과 연결될 수 있습니다.*

## 흔히 발생하는 실수를 피하세요

| 실수 | 문제 | 수정 |
|---------|---------|------|
| **신체** | 30개 이상의 속성을 가진 하나의 엔터티 | 집중된 엔터티로 분할 |
| **식별자 누락** | 인스턴스를 계산하거나 그룹화할 수 없음 | 고유 식별자 속성 추가 |
| **모호한 관계 이름** | `relatedTo`, `hasLink` | 특정 동사를 사용하세요: `prescribes`, `enrolledIn` |
| **원형 일대일** | A → B 및 B → A 모두 1:1 | 아마도 동일한 엔터티일 것입니다. 병합 |
| **오버모델링** | 모든 내부 테이블은 엔터티가 됩니다 | 스키마가 아닌 사용자가 쿼리할 내용을 모델링하세요 |

## 설명을 사용해야 하는 경우

모든 엔터티 유형, 속성 및 관계에는 선택적 **설명**이 있을 수 있습니다. 다음과 같은 경우에 사용하세요.

- 이름만으로는 모호합니다. (`status`는 많은 의미를 가질 수 있습니다.)
- 개념은 도메인별로 다릅니다 (`formulary`, `SKU`, `yield`)
- 자연어 쿼리 해석을 안내하고 싶으신 분

## 주요 내용

- 인간의 이름: 단수명사, 낙타표기법, 동사구
- 하나의 엔터티, 하나의 개념 — 과부하된 엔터티 분할
- 안정적이고 고유하며 의미 있는 식별자를 선택하세요.
- 외래 키 열이 아닌 이름을 사용한 모델 관계
- 적절한 집계를 활성화하려면 카디널리티를 올바르게 설정하세요.
- 이름이 모호한 경우 설명 추가

```quiz
Q: Person 엔터티에는 급여, PatientId, CourseGrade 및 accountBalance 속성이 있습니다. 어떤 디자인 패턴을 적용해야 할까요?
- 식별자 속성을 추가합니다.
- 모든 속성을 설명 필드에 병합
- 별도의 엔터티 유형(직원, 환자, 학생, 고객)으로 분할하고 관련 [correct]
- 단순하게 유지하기 위해 하나의 속성만 남기고 모두 제거합니다.
> 개체가 관련 없는 속성을 축적하면 "신 개체"가 됩니다. 해결책은 각 개념을 고유한 엔터티 유형으로 분리하고 필요한 경우 관계와 연결하는 것입니다.
```
