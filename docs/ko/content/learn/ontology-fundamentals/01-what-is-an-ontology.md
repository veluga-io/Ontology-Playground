## 그래프로 생각하기

당신이 커피숍을 묘사하고 있다고 상상해 보세요. 상점, 제품, 고객, 주문 등 **사물**과 이들 사이의 **연결**에 대해 이야기할 수 있습니다. 즉, 고객이 *주문*하고, 주문이 *제품을 *포함*하고, 매장이 *제품*을 *재고*합니다.

**온톨로지**는 도메인에 있는 사물의 유형과 이들이 서로 어떻게 관련되어 있는지를 정확하게 설명하는 공식적인 방법입니다. 이는 데이터 자체가 아니라 데이터의 *모양*에 대한 청사진입니다.

## 엔터티, 속성 및 관계

모든 온톨로지는 세 가지 구성 요소로 구성됩니다.

| 개념 | 의미 | 예 |
|---------|--------------|---------|
| **엔티티 유형** | 사물의 카테고리 | `Customer`, `Product`, `Store` |
| **재산** | 법인에 관한 사실 | `Customer.name`, `Product.price` |
| **관계** | 엔터티 간의 연결 | `Customer → places → Order` |

속성에는 텍스트, 숫자, 날짜, 부울 등 **유형**이 있으며 모든 엔터티에는 각 인스턴스를 고유하게 구별하는 하나 이상의 **식별자 속성**(예: 고객 ID)이 필요합니다.

## 온톨로지가 중요한 이유

온톨로지가 없으면 데이터는 단지 테이블과 열일 뿐입니다. 하나를 사용하면 시스템은 "수익"이 `Order.totalAmount`로 그룹화된 `Store.city`의 합계라는 것을 이해할 수 있습니다. 온톨로지가 이러한 개념이 어떻게 연결되는지 알려주기 때문입니다.

이것이 **의미론적 데이터 모델**의 기초입니다. SQL을 직접 작성하는 대신 일반 언어로 원하는 것을 설명하면 시스템이 온톨로지를 사용하여 올바른 쿼리를 생성합니다.

## 컨셉부터 코드까지

온톨로지는 일반적으로 클래스, 속성 및 관계를 설명하기 위한 XML 기반 표준인 **RDF/OWL**로 표현됩니다. 하지만 XML을 직접 작성할 필요는 없습니다. [온톨로지 디자이너](#/designer)와 같은 도구를 사용하면 시각적으로 온톨로지를 구축하고 유효한 RDF를 내보낼 수 있습니다.

## 주요 내용

- 온톨로지는 도메인의 **사물 유형**과 **사물 유형**을 정의합니다.
- 데이터가 아닌 스키마입니다. 내용이 아닌 모양을 설명합니다.
- 의미론적 쿼리가 가능합니다. 자연어로 질문하고 구조화된 답변을 얻습니다.
- 표준 형식은 **RDF/OWL**이지만 JSON 표현으로도 작업할 수 있습니다.

```quiz
Q: 다음 중 온톨로지의 구성 요소가 아닌 것은 무엇입니까?
- 엔터티 유형
- 부동산
- SQL 쿼리 [correct]
- 관계
> 온톨로지는 엔터티 유형, 속성 및 관계로 구축됩니다. SQL 쿼리는 데이터를 검색하는 방법입니다. 이는 온톨로지 정의 자체의 일부가 아닙니다.
```

```quiz
Q: 식별자 속성의 목적은 무엇입니까?
- 엔터티의 색상을 저장하려면
- 엔터티의 각 인스턴스를 고유하게 구별하려면 [correct]
- 두 엔터티를 함께 연결하려면
- 데이터 형식을 정의하려면
> 식별자 속성(예: 고객 ID)은 엔터티 유형 내의 각 인스턴스를 고유하게 식별하므로 시스템이 올바르게 계산, 그룹화 및 조인할 수 있습니다.
```
