## 케어 전달 재단

의료 서비스 제공은 세 가지 개념을 중심으로 진행됩니다.

- **환자** — 누가 치료를 받고 있나요?
- **제공자** — 누가 진료를 제공합니까?
- **예약** — 진료는 언제 어디서 이루어지나요?

이 세 가지 엔터티는 의료 일정 및 제공을 포착합니다. 모든 진단과 치료는 예약을 통해 이루어집니다.

## 엔터티 정의

### 환자

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `patientId` | 문자열 | `mrn` |
| `dateOfBirth` | 문자열 | |
| `bloodType` | 날짜 | |
| `allergies` | 문자열 | |
| ✓ | 문자열 | |

`mrn`(의료 기록 번호)는 병원의 내부 식별자입니다. `patientId`는 온톨로지 식별자로 사용되는 반면, `mrn`는 EHR 시스템에 매핑되는 도메인별 속성입니다.

### 제공자

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `providerId` | 문자열 | `name` |
| `specialty` | 문자열 | |
| `licenseNumber` | 문자열 | |
| `department` | 문자열 | |
| ✓ | 문자열 | |

`specialty` 및 `department` 속성을 사용하면 임상 도메인별로 제공자를 필터링할 수 있으며 이는 추천 및 라우팅 쿼리에 필수적입니다.

### 약속

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `appointmentId` | 문자열 | `scheduledTime` |
| `duration` | 날짜/시간 | |
| `type` | 정수(분) | |
| `status` | 문자열 | |
| ✓ | 문자열 | |

`duration` 속성은 분 단위의 정수를 사용하므로 일정 계산 및 활용도 분석이 가능합니다.

## 관계

- **has_appointment** — `Patient` → `Appointment`(일대다) 환자는 시간이 지남에 따라 많은 약속을 가질 수 있습니다.

- **본다** — `Provider` → `Appointment`(일대다) 공급자가 많은 약속을 처리합니다.

> **공유 엔터티 패턴:** 약속은 환자와 제공자 *모두*에 연결됩니다. 두 개의 독립적인 개체가 상호 작용하는 만남의 장소입니다. 이 패턴은 두 배우가 동일한 이벤트에 참여할 때마다 일반적입니다.

## 우리가 배운 것

- **공유 엔터티**(약속)는 두 개의 독립 행위자(환자, 제공자)를 연결합니다.
- **기간 속성**은 단위(분, 시간, 일)와 함께 정수를 사용합니다.
- **도메인별 식별자**(MRN)는 온톨로지 식별자(patientId)와 공존합니다.
- 일정 삼각형(환자-약속-제공자)은 의료 기반입니다.

```quiz
Q: 약속이 단 한 사람이 아닌 환자와 제공자 모두에 연결되는 이유는 무엇입니까?
- 그래프를 더욱 완성도 있게 보이도록 하기 위해
- 약속은 공유 엔터티이기 때문에 두 행위자 간의 상호 작용 지점을 나타냅니다. [correct]
- 모든 엔터티에는 최소한 두 개의 관계가 있어야 하기 때문입니다.
- 환자와 제공자의 속성이 동일하기 때문에
> 약속은 본질적으로 환자와 의료 제공자 모두가 참여하는 공동 이벤트입니다. 두 관계를 모델링하면 전체 일정 계획을 파악하고 "환자의 다음 방문은 언제입니까?"라는 두 가지 관점에서 쿼리를 수행할 수 있습니다. 또는 "이 의료 제공자는 하루에 몇 명의 환자를 진료합니까?"
```
