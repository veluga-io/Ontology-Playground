## 치료 체인

의료 퍼즐의 마지막 조각은 진단에 대한 치료 반응인 **처방**입니다. 이로써 예약 → 진단 → 치료라는 진료 주기가 종료됩니다.

## 처방 엔터티

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `rxNumber` | 문자열 | `medication` |
| `dosage` | 문자열 | |
| `frequency` | 문자열 | |
| `refillsRemaining` | 문자열 | |
| ✓ | 정수 | |

식별자는 약국 표준 식별자인 `rxNumber`(처방전 번호)입니다. `refillsRemaining` 정수는 리필 추적 및 약물 준수 모니터링을 가능하게 합니다.

## 새로운 관계

- **치료 대상** — `Diagnosis` → `Prescription`(일대다) 진단은 여러 처방으로 이어질 수 있습니다(예: 동일한 상태에 대해 여러 약물).

- **처방** — `Provider` → `Prescription` (일대다) 의료 제공자가 환자를 위해 처방전을 작성합니다.

> **케어 체인:** 이제 전체 경로는 `Patient → Diagnosis → Prescription`이며, `Provider`는 모든 단계(약속 확인, 진단, 처방전 작성)에서 연결됩니다. 이는 실제 임상 작업 흐름을 반영합니다.

*완전한 의료 온톨로지: 5개 엔터티, 6개 관계. 케어 체인은 환자에서 진단을 거쳐 처방까지 이어집니다.*

## 완전한 모델이 가능하게 하는 것

| 질문 | 그래프 경로 |
|---|---|
| 어떤 환자에게 처방전 리필이 필요한가요? | 환자 → 진단 → 처방(refillsRemaining=0) |
| 어떤 의료 제공자가 가장 많은 약을 처방합니까? | 제공자 → 처방(횟수) |
| 아직 치료법이 없는 심각한 진단은 무엇입니까? | 무증상 진단(심각도=심각) → 처방 |
| 어떤 전문가가 자신이 처방하는 질환을 진단합니까? | 제공자 → 진단 AND 제공자 → 처방전 |

## GQL 쿼리 예

처방전이 부족한 중증환자를 찾아보세요.

```gql
MATCH (p:Patient)-[:diagnosed_with]->(d:Diagnosis)-[:treated_by]->(rx:Prescription)
WHERE d.severity = 'severe' AND rx.refillsRemaining <= 1
RETURN p.patientId, d.description, rx.medication, rx.refillsRemaining
```

## 우리가 만든 것

| 단계 | 추가된 엔터티 | 누적 | 핵심 개념 |
|---|---|---|---|
| 1 | 환자, 제공자, 예약 | 3 | 공유 엔터티, 예약 |
| 2 | 진단 | 4 | 표준화된 코드, 이중 연결 |
| 3 | 처방 | 5 | 케어 체인, 치료 추적 |

## 주요 내용

1. **공유 엔터티**(예약, 진단)는 여러 행위자를 연결합니다.
2. **표준화된 코드**(ICD, Rx)는 시스템 간 상호 운용성을 가능하게 합니다.
3. **케어 체인**(환자 → 진단 → 처방) 모델 임상 워크플로우
4. **제공자는 모든 단계에서 연결** - 의료 서비스 제공의 중심 역할을 반영
5. **정수 속성**(refillsRemaining, 기간)은 운영 쿼리를 활성화합니다.

```quiz
Q: 공급자 엔터티는 전체 의료 온톨로지에 어떻게 연결됩니까?
- 공급자는 약속에만 연결됩니다.
- 공급자는 진료의 모든 단계에서 자신의 역할을 반영하여 예약, 진단 및 처방에 연결됩니다. [correct]
- 제공자가 환자에게 직접 연결
- 공급자는 처방전에만 연결됩니다.
> 제공자는 이 온톨로지에서 가장 연결된 개체입니다. 약속을 확인하고, 진단하고, 처방전을 작성합니다. 이는 의료 제공자가 의료 제공 체인의 모든 단계에 참여하는 실제 워크플로우를 반영합니다.
```
