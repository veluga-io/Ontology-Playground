## 위치 컨텍스트 추가

상거래는 진공 상태에서 발생하는 것이 아니라 **장소**에서 발생합니다. 지리적 엔터티를 추가하면 다음과 같은 질문에 답할 수 있습니다.

- "남서부 지역 전체 매출은 얼마였나요?"
- "북동쪽에는 어떤 매장이 주문량이 감소하고 있나요?"
- "서남부 지역은 콜드체인 물류가 필요한가요?"

## 지역

지역은 상점과 창고를 포함하는 광범위한 지리적 영역을 나타냅니다.

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `regionId` | 문자열 | `regionName` |
| `timezone` | 문자열 | |
| `coldChainRequired` | 문자열 | |
| ✓ | 부울 | |

`coldChainRequired` 속성은 **부울 속성**의 좋은 예입니다. 예/아니오 비즈니스 규칙(이 지역에 냉장 배송이 필요한가요?)을 문서에 묻어두는 대신 쿼리 가능한 속성으로 캡처합니다.

## 매장

매장은 고객이 주문하는 실제 소매점입니다.

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `storeId` | 문자열 | `storeName` |
| `address` | 문자열 | |
| ✓ | 문자열 | |

## 새로운 관계

- **OrderFulfilledToRegion** — `Order` → `Region`(다대일) 각 주문은 하나의 지역에서 처리됩니다. 이를 통해 지역별 매출 분석이 가능해집니다.

- **StoreInRegion** — `Store` → `Region`(다대일) 각 매장은 하나의 지역에 속합니다. 순서 관계와 결합하면 지역에 대한 두 가지 경로가 생성됩니다. 이는 다양한 분석 관점에 유용합니다.

## 지리적 계층

**Store** → **Region**이 **지리적 계층 구조**를 형성하는 패턴을 확인하세요. 더 자세한 온톨로지에서는 도시, 주 또는 국가 수준을 추가할 수 있습니다. 핵심 원칙은 다음과 같습니다.

> 계층 구조의 각 수준은 다대일 관계를 통해 다음 수준에 연결됩니다. 이를 통해 자동 롤업이 가능합니다. 매장 수준 데이터가 지역 수준 합계로 집계됩니다.

*7가지 엔터티 유형. 지역 및 상점은 지리적 컨텍스트를 추가합니다. 이제 주문이 고객(구매자)과 지역(이행된 곳)에 어떻게 연결되는지 확인하세요.*

## 우리가 배운 것

- **지리적 실체**는 위치 기반 분석을 가능하게 합니다.
- **부울 속성**(예: `coldChainRequired`)은 비즈니스 규칙을 캡처합니다.
- **계층**을 통해 데이터를 세분화된 데이터에서 집계 데이터로 롤업할 수 있습니다.
- 엔터티는 다양한 엔터티와 여러 관계를 가질 수 있습니다. 주문은 고객, 제품(OrderLine을 통해) 및 지역에 연결됩니다.

```quiz
Q: Store에 "지역" 텍스트 속성을 추가하는 대신 지역을 별도의 엔터티로 모델링하는 것이 유용한 이유는 무엇입니까?
- 저장 공간이 절약됩니다.
- 롤업 쿼리를 활성화하고 지리 데이터에 대한 단일 진실 소스를 유지합니다. [correct]
- 온톨로지 다이어그램을 더 보기 좋게 만듭니다.
- Fabric IQ가 필요합니다.
> 지역을 별도의 엔터티로 모델링하면 매장 → 지역이라는 계층 구조가 생성됩니다. 이를 통해 IQ는 지역별 데이터("북동부의 총 매출")를 집계하고 지역 메타데이터가 모든 매장에 중복되지 않고 한 번 정의되도록 할 수 있습니다.
```

다음으로 주문이 실제로 배송되는 방식을 모델링하겠습니다.
