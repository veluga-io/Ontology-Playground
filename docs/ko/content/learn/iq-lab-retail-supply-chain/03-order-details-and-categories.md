## 다대다 문제

1단계에서는 다대다 관계를 통해 Order를 Product에 직접 연결했습니다. 이는 "이 주문에 어떤 제품이 있었나요?"와 같은 간단한 쿼리에 적합합니다. — 하지만 **수량**과 **라인 합계**는 어떻습니까?

직접적인 다대다 관계는 속성을 전달할 수 없습니다. 고객 A가 제품 X 3개를 주문하고 고객 B가 1개를 주문한 경우 수량은 어디에 있습니까? 주문에 없고(여러 제품이 있음) 제품에도 없습니다(여러 주문에 나타남).

## 연결 엔터티 패턴

해결책은 **연결 엔터티**입니다. 이 엔터티 유형은 다른 두 엔터티 사이에 위치하며 연결별 속성을 전달합니다.

**OrderLine**은 주문을 제품에 연결하고 다음을 보유합니다.

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `orderLineId` | 문자열 | `quantity` |
| `lineTotal` | 정수 | |
| ✓ | 십진수(USD) | |

### 새로운 관계

- **OrderHasLineItem** — `Order` → `OrderLine`(일대다) 각 주문에는 하나 이상의 품목이 있습니다.

- **OrderLineReferencesProduct** — `OrderLine` → `Product`(다대일) 각 품목은 정확히 하나의 제품을 참조합니다.

이제 순회는 `Order` → `OrderLine` → `Product`이며 각 라인 항목에는 자체 `quantity` 및 `lineTotal`가 포함됩니다.

> **디자인 패턴:** 다대다 관계에 속성(수량, 가격, 날짜)이 필요할 때마다 연결 엔터티를 도입하세요. 이는 관계형 데이터베이스의 연관 테이블과 동등한 온톨로지입니다.

## 카테고리로 정리하기

제품은 단독으로 존재하는 경우가 거의 없으며 '냉동식품', '가정용품', '전자제품'과 같은 **카테고리**에 속합니다. ProductCategory 엔터티를 추가하면 제품을 그룹화하고 "가장 많은 반품이 발생한 카테고리는 무엇입니까?"와 같은 질문에 답할 수 있습니다.

**제품 카테고리**:

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `categoryId` | 문자열 | `categoryName` |
| ✓ | 문자열 | |

### 새로운 관계

- **ProductInCategory** — `Product` → `ProductCategory`(다대일) 각 제품은 정확히 하나의 카테고리에 속합니다.

## 2단계 그래프

<ontology-embed id="official/iq-lab-retail-step-2" diff="official/iq-lab-retail-step-1" height="400px"></ontology-embed>

*5가지 관계로 연결된 5가지 엔터티 유형. OrderLine은 주문과 제품 사이의 다리 역할을 하며 수량 데이터를 전달합니다. ProductCategory는 제품을 그룹화합니다.*

## 우리가 배운 것

- **엔티티 연결**은 다대다 속성 문제를 해결합니다.
- 관계에 자체 데이터가 필요한 경우 이를 엔터티로 모델링합니다.
- **계층**(제품 → ProductCategory)은 롤업 쿼리를 활성화합니다.
- 그래프가 성장하고 있습니다. 각각의 새로운 엔터티는 기존 엔터티에 연결됩니다.

```quiz
Q: 언제 직접적인 관계 대신 연결 엔터티(예: OrderLine)를 도입해야 합니까?
- 엔터티 유형이 3개 이상인 경우
- 두 엔터티 간의 관계에 고유한 속성이 필요한 경우 [correct]
- 두 엔터티 유형 모두 식별자 속성을 갖는 경우
- 엔터티가 다른 네임스페이스에 있는 경우
> 다대다 관계가 자체 데이터(예: 수량 또는 줄 합계)를 전달해야 하는 경우 연결 엔터티가 필요합니다. 직접적인 관계는 속성을 보유할 수 없습니다.
```
