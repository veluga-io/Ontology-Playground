## 전달 파이프라인

고객은 주문을 합니다. 그런데 해당 주문이 고객에게 어떻게 전달되나요? 이행 계층은 주문을 물리적 물류 인프라에 연결합니다.

- **배송** — 배송 기록
- **운송업체** — 물류 제공업체(FedEx, UPS 등)
- **창고** — 제품을 보관하고 배송하는 주문 처리 센터

## 새로운 관계

세 가지 새로운 관계가 물류 엔터티를 연결합니다.

- **ShipmentFulfillsOrder** — `Shipment` → `Order`(다대일) 각 배송은 하나의 주문을 이행합니다. 여러 배송으로 동일한 주문을 처리할 수 있습니다(분할 배송).

- **ShipmentByCarrier** — `Shipment` → `Carrier`(다대일) 각 배송은 하나의 배송업체에서 처리됩니다.

- **창고에서 출발한 배송** — `Shipment` → `Warehouse`(다대일) 각 배송은 하나의 창고에서 출발합니다.

## 허브 패턴

**배송**은 **허브 엔터티** 역할을 하며 주문, 운송업체 및 창고에 동시에 연결됩니다. 이는 여러 개념을 연결하는 트랜잭션 또는 이벤트 엔터티에 대한 일반적인 패턴입니다.

```
Carrier ← Shipment → Order → Customer
              ↓
          Warehouse
```

Carrier에서 Shipment, Order, Customer까지의 단일 그래프 순회는 "어떤 고객이 CarrierX로부터 배송을 받았습니까?"라고 대답합니다.

## 4단계 그래프

*풍부한 연결된 그래프를 형성하는 10가지 엔터티 유형. 배송은 물류 계층(운송업체, 창고)과 상거래 계층(주문, 고객)을 연결합니다. 그래프를 통해 모든 창고에서 모든 고객으로 이동할 수 있습니다.*

## 우리가 배운 것

- **허브 엔터티**(예: 배송)는 여러 도메인을 연결합니다.
- 물류 계층은 상거래 계층을 확장합니다. 기존 엔터티를 수정할 필요가 없습니다.
- 그래프 순회를 통해 도메인 간 쿼리가 자연스러워집니다. "남서부 지역으로 배송되는 창고는 어디입니까?" SQL 조인이 필요하지 않습니다.
- 온톨로지는 이제 10개의 관계를 가진 10개의 엔터티입니다. — 성장하고 있지만 여전히 읽을 수 있습니다.

```quiz
Q: 배송 엔터티는 온톨로지에서 어떤 역할을 합니까?
- Order 엔터티를 대체합니다.
- 물류 레이어와 커머스 레이어를 연결하는 허브 역할을 합니다. [correct]
- 고객 주소를 저장합니다.
- Warehouse와 Carrier 간의 카디널리티를 정의합니다.
> 배송은 여러 도메인을 연결하는 허브 엔터티입니다. 주문을 물류 인프라(운송업체, 창고)에 연결하여 기존 엔터티를 수정하지 않고도 도메인 간 쿼리가 가능합니다.
```
