## 기초

모든 소매 시스템은 세 가지 핵심 개념으로 시작됩니다.

- **고객** — 누가 구매하나요?
- **주문** — 어떤 거래가 발생했나요?
- **제품** — 무엇을 구매했습니까?

이 세 가지 엔터티 유형은 온톨로지의 핵심을 형성합니다. 이후 단계에서 추가하는 모든 항목은 다시 연결됩니다.

## 엔터티 유형 정의

각 엔터티 유형에는 다음이 필요합니다.

1. A **이름** — 단수형, 설명형(예: `Customer`, `Customers` 또는 `tbl_cust` 아님)
2. **식별자 속성** — 각 인스턴스의 고유 키
3. **속성** — 각 인스턴스를 설명하는 속성

### 고객

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `customerId` | 문자열 | `name` |
| `email` | 문자열 | |
| `loyaltyTier` | 문자열 | |
| `lifetimeValue` | 문자열 | |
| ✓ | 십진수(USD) | |

`customerId`는 각 고객을 고유하게 식별합니다. `loyaltyTier` 및 `lifetimeValue`와 같은 속성은 원본 데이터베이스의 잠재적으로 암호화된 열 이름에 매핑되는 비즈니스에 의미 있는 이름입니다.

### 주문

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `orderId` | 문자열 | `orderDate` |
| `status` | 날짜/시간 | |
| `totalAmount` | 문자열 | |
| ✓ | 십진수(USD) | |

### 제품

| 부동산 | 유형 | 식별자? |
|---|---|---|
| `productId` | 문자열 | `name` |
| `unitCost` | 문자열 | |
| `discountPercent` | 십진수(USD) | |
| ✓ | 소수점(%) | |

## 관계로 연결하기

엔터티 자체는 단지 격리된 테이블일 뿐입니다. **관계**는 이를 연결된 그래프로 바꿉니다.

- **OrderPlacedByCustomer** — `Order` → `Customer`(다대일) 각 주문은 정확히 한 명의 고객에 의해 이루어지지만 고객은 여러 주문을 할 수 있습니다.

- **OrderContainsProduct** — `Order` → `Product`(다대다) 주문에는 여러 제품이 포함될 수 있으며 제품은 여러 주문에 나타날 수 있습니다.

### 카디널리티 문제

카디널리티는 시스템에 계산 및 집계 방법을 알려줍니다.

| 카디널리티 | 의미 | 예 |
|---|---|---|
| 일대일 | 양쪽에 정확히 하나씩 | 직원 → 배지 |
| 일대다 | 한 부모, 많은 자녀 | 고객 → 주문 |
| 다대일 | 많은 자녀, 한 부모 | 주문 → 고객 |
| 다대다 | 제한 없음 | 주문 ⇔ 상품 |

올바른 카디널리티를 선택하면 "각 고객이 몇 건의 주문을 했습니까?"와 같은 쿼리가 보장됩니다. 올바른 개수를 반환합니다.

*고객, 주문, 제품은 두 개의 관계로 연결됩니다. 이는 다른 모든 것이 구축되는 기초입니다.*

## 우리가 배운 것

- 모든 엔터티 유형에는 식별자 속성이 필요합니다.
- 내부 열 이름이 아닌 비즈니스에 의미 있는 이름을 사용하세요.
- 관계에는 데이터 계산 방법에 영향을 미치는 카디널리티가 있습니다.
- 세 개의 개체라도 유용한 연결 그래프를 만듭니다.

```quiz
Q: 고객은 여러 주문을 할 수 있지만 각 주문은 한 고객에게 속합니다. 이것은 어떤 카디널리티입니까?
- 일대일
- 다대다
- 일대다 [correct]
- 다대일
> 고객의 관점에서 볼 때 이는 일대다입니다. 즉, 한 고객이 여러 주문을 받을 수 있습니다. Order의 관점에서 볼 때 이는 다대일입니다. 관계는 일대다 카디널리티를 사용하여 고객 → 주문으로 정의됩니다.
```

다음으로 주문에 세부 정보를 추가하고 제품을 카테고리로 구성하겠습니다.
